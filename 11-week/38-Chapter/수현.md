# 38장 - 브라우저의 렌더링 과정

브라우저가 HTML, CSS, 자바스크립트로 작성된 텍스트 문서를 어떻게 **파싱**하여 브라우저에 **렌더링**하는지 살펴보는 챕터

**렌더링 과정**

1. 브라우저는 HTML, CSS, 자바스크립트, 이미지, 폰트 파일 등 렌더링에 필요한 리소스를 요청하고 서버로부터 응답을 받는다.
2. 브라우저의 렌더링 엔징은 서버로부터 응답된 HTML과 CSS를 파싱하여 DOM과 CSSOM을 생성하고 이들을 결합하여 렌더 트리를 생성한다.
3. 브라우저의 자바스크립트 엔진은 서버로부터 응답된 자바스크립트를 파싱하여 AST를 생성하고 바이트코드로 변환하여 실행한다. 이때 자바스크립트는 DOM API를 통해 DOM이나 CSSOM을 변경할 수 있다. 변경된 COM과 CSSOM은 다시 렌더 트리로 결합된다.
4. 렌더 트리를 기반으로 HTML요소의 레이아웃을 계산하고 브라우저 화면에 HTML요소를 페인팅한다.

## 요청과 응답

브라우저의 핵심 기능

: 필요한 리소스를 서버에 요청하고 서버로부터 응답받아 브라우저에 시각적으로 렌더링하는 것

서버에 요청을 전송하기 위해 브라우저는 주소창을 제공함. 브라우저의 주소창에 url을 입력하고 엔터를 누르면 url의 호스트 이름이 dns를 통해 ip주소로 변환되고 이 ip주소를 갖는 서버에게 요청을 전송함.

요청과 응답은 개발자 도구 network 패널에서 확인할 수 있음

## HTML 파싱과 DOM 생성

브라우저 렌더링 엔진은 다음과 같은 과정을 통해 응답받은 HTML문서를 파싱하여 브라우저가 이해할 수 있는 자료구조인 DOM을 생성함

1. 서버에 존재하던 HTML 파일이 브라우저의 요청에 의해 응답된다. 이때 서버는 브라우저가 요청한 HTML 파일을 읽어 메모리에 저장한 다음 메모리에 저장된 바이트(2진수)를 인터넷을 공유하여 응답한다.
2. 브라우저는 서버가 응답한 HTML 문서를 화면에 렌더링한다. 그리고 응답받은 바이트 형태의 HTML 문서를 meta 태그의 charset 어트리뷰트(charset=UTF-8이라고 되어 있음)에 의해 지정된 인코딩 방식( 등)을 기준으로 문자열로 변환한다. 참고로 meta 태그의 charset 어트리뷰트는 이제 content-type: text/html; charset=utf-8과 같이 응답 헤더인 response header에도 존재한다.
3. 문자열로 변환된 HTML 문서를 읽어 들인다. 브라우저는 문자열 HTML 문서를 위에서 아래로 한 줄씩 순차적으로 읽어 들이면서 들어오는 내용에 따라 문서 노드, 요소 노드, 텍스트 노드로 분리한다.
4. 문자열을 쪼개고 변환하여 노드(node)를 생성한다. 또한 노드 간의 관계에 의해 트리 자료구조를 생성한다. 노드와 노드의 관계는 부모 노드와 자식 노드로 이루어진다.
5. HTML 문서는 HTML 요소의 집합으로 이루어지며 HTML 요소는 중첩 관계를 갖는다. 즉 HTML 요소의 렌더링 영역(시각적 표현 범위)은 태그 사이에 속하는 텍스트뿐만 아니라 HTML 요소를 포함할 수 있다. HTML 요소 간에는 중첩 관계에 의해 부모 요소와 자식 요소의 관계가 형성된다. 이러한 HTML 요소 간의 중첩 관계에 의해 부자 관계의 트리 자료구조를 DOM(Document Object Model)이라고 부른다.

**DOM은 HTML문서를 파싱한 결과물**

## CSS 파싱과 CSSOM 생성

렌더링 엔진은 dom을 생성해나가다가 css를 로드하는 link나 style태그를 만나면 dom생성을 일시 중단함.

css를 html과 동일한 파싱 과정을 거치며 해석하여 CSSOM을 생성

그 과정이 끝나면 다시 dom생성을 시작함.

## 렌더 트리 생성

dom과 cssom은 렌더 트리로 결합됨

렌더 트리는 렌더링을 위한 트리 구조의 자료구조

브라우저의 렌더링 과정은 반복해서 실행될 수 있는데 이런 경우 반복해서 레이아웃 계산과 페인팅이 재차 실행됨

- 자바스크립트에 의한 노드 삭제
- 브라우저 창의 리사이징에 의한 뷰포트 크기 설정
- html요소의 레이아웃에 변경을 발생시키는 스타일 변경

⇒ 이러한 것들은 다시 실행하는 리렌더링은 비용이 많이 드는 성능에 악역향을 주는 작업이므로 가급적 리렌더링이 빈번하게 발생하지 않도록 주의할 필요가 있음.

## 자바스크립트 파싱에 의한 HTML 파싱 중단

브라우저는 웹페이지를 만들 때 HTML을 위에서 아래로 순차적으로 파싱하면서 DOM을 만든다.

그런데 중간에 `<script>` 태그를 만나면 HTML 파싱을 멈춘다.

왜? 자바스크립트가 HTML 구조(DOM)를 변경할 수도 있기 때문

즉, JS를 먼저 실행해야 안전하게 다음 HTML을 읽을 수 있으므로 브라우저는 HTML 파싱을 잠시 멈추고 JS 실행을 기다린다.

## script 태그의 async/defer 어트리뷰트

자바스크립트 파싱에 의한 dom 생성 중단되는 문제를 근본적으로 해결하기 위해 html5부터 script태그에 async, defer 어트리뷰트가 추가됨

비동기적 실행

| 속성            | 다운로드 시점           | 실행 시점                    | HTML 파싱 중단? | 실행 순서                                     |
| --------------- | ----------------------- | ---------------------------- | --------------- | --------------------------------------------- |
| **없음 (기본)** | 즉시                    | 다운로드 후 바로 실행        | **중단됨**      | HTML 순서대로                                 |
| **async**       | HTML 파싱과 동시에 병렬 | 다운로드 끝나는 즉시 실행    | **중단됨**      | **순서 보장 X** (다운로드 빨리 끝난 순서대로) |
| **defer**       | HTML 파싱과 동시에 병렬 | **HTML 파싱이 끝난 뒤 실행** | 중단 없음       | **HTML 순서대로**                             |
